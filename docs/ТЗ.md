# Техническое задание: Clinkon1C

> **Репозиторий:** github.com/*/Clinkon1C  
> **Язык интерфейса:** Русский  
> **Язык реализации:** C# (.NET 8 LTS)  
> **UI-фреймворк:** Terminal.Gui  
> **Формат дистрибутива:** Self-contained `.exe` (без установки .NET)  
> **Лицензия:** MIT  
> **Минимальная ОС:** Windows 10 / Windows Server 2012 R2  

---

## 1. Назначение

Clinkon1C — консольная утилита с псевдографическим интерфейсом для администраторов систем 1С на Windows и RDP-серверах. Инструмент предназначён **исключительно для администраторов** и не должен быть доступен рядовым пользователям.

---

## 2. Архитектура

### 2.1 Модульность

Каждый верхний уровень дерева — отдельный модуль (`.cs` файл), реализующий общий интерфейс `IModule`. Главное дерево динамически загружает зарегистрированные модули. Добавление нового модуля = новый файл + регистрация в дереве.

```
src/
├── Core/
│   ├── IModule.cs              — интерфейс модуля
│   ├── TreeBuilder.cs          — построение дерева
│   ├── ProfileFinder.cs        — поиск профилей пользователей
│   ├── ProcessHelper.cs        — работа с процессами 1С
│   ├── RestartManagerHelper.cs — определение блокирующих процессов (WinAPI)
│   ├── SafeDelete.cs           — безопасное удаление (Backup, защита файлов)
│   ├── Registry.cs             — работа с реестром (настройки, исключения)
│   └── Logger.cs               — лог операций утилиты
├── Modules/
│   ├── Cache/
│   │   └── CacheModule.cs      — модуль кэша (MVP)
│   ├── Logs/
│   │   └── LogsModule.cs       — модуль логов (Phase 2)
│   └── Templates/
│       └── TemplatesModule.cs  — модуль шаблонов (Phase 2)
├── UI/
│   ├── MainTree.cs             — главное дерево
│   ├── ActionBar.cs            — панель действий
│   └── Dialogs.cs              — диалоги подтверждения
└── Program.cs
```

### 2.2 Фазы разработки

| Фаза | Содержимое |
|---|---|
| **MVP (Phase 1)** | Главное дерево + модуль Кэш |
| Phase 2 | Модуль Логи + модуль Шаблоны |
| Phase 3 | Процессы 1С, серверные службы, RAC/RAS |
| Phase 4 | Silent-режим, CLI-параметры, планировщик |

### 2.3 Хранение настроек

Все настройки утилиты хранятся в реестре Windows:

| Ключ | Содержимое |
|---|---|
| `HKCU:\Software\Clinkon1C\Settings` | Основные настройки (пороги, путь backup) |
| `HKCU:\Software\Clinkon1C\Exclusions` | Исключения (пользователи, базы) |

## 4. Интерфейс — дерево

### 4.1 Общая концепция

Единая панель в виде дерева (как в файловых менеджерах). Вся навигация, выделение и действия — из одного экрана без вложенных меню.

### 4.2 Цветовая схема

| Элемент | Цвет |
|---|---|
| Заголовки, рамки | Cyan |
| Активные узлы | White |
| Недоступные узлы | DarkGray |
| Выделенные (пробел) | Yellow |
| Успех | Green |
| Предупреждение | Yellow |
| Ошибка | Red |
| Dry Run | Magenta |
| Исключения | DarkCyan + `[исключён]` |

### 4.3 Навигация

| Клавиша | Действие |
|---|---|
| ↑ ↓ | Перемещение по дереву |
| → / Enter | Раскрыть узел |
| ← | Свернуть / подняться |
| Пробел | Выделить узел (и всё дерево под ним) |
| Tab | Переключить вид (по пользователю / по базе) |
| D | Dry Run для выделенного |
| Del | Удалить выделенное |
| R | Обновить дерево |
| Esc | Снять выделение |
| F1 | Помощь |

### 4.4 Панель действий (нижняя строка)

```
──────────────────────────────────────────────────────────────
[Пробел] Выделить  [D] Dry Run  [Del] Удалить  [Tab] Сменить вид
Выделено: 2 объекта  │  14.6 GB  │  Кэш: 40.2 GB  Логи: 3.1 GB
──────────────────────────────────────────────────────────────
```

### 4.5 Уровни подтверждения

| Уровень | Пример | Подтверждение |
|---|---|---|
| Обычная | Кэш одной базы одного пользователя | `[Y / N]` |
| Опасная | Кэш всех баз всех пользователей | `[Y / N]` + предупреждение |
| Критическая | Журналы регистрации, серверные службы | Ввод слова `УДАЛИТЬ` |

## 5. Поиск профилей пользователей

Профили ищутся через реестр (не только `C:\Users\*`):

```
HKLM:\SOFTWARE\Microsoft\Windows NT\CurrentVersion\ProfileList
```

Поддерживаются:
- Локальные профили
- Roaming Profiles
- FSLogix-контейнеры
- Redirected AppData

При недоступности сетевого пути — пропуск с записью `WARN` в лог.

## 6. Модуль Кэш (MVP)

### 6.1 Расположение кэша

Для каждого пользователя и каждой найденной версии платформы:

| Путь | Содержимое |
|---|---|
| `%LOCALAPPDATA%\1C\1cv8\{uuid}\` | Основной кэш базы |
| `%APPDATA%\1C\1cv8\` | Пользовательские настройки платформы |
| `%TEMP%\1C\` | Временные файлы платформы |

### 6.2 Определение версий платформы

Все установленные версии ищутся через:
- Реестр: `HKLM:\SOFTWARE\1C\1Cv8\`
- Папки профиля: `%LOCALAPPDATA%\1C\1cv8\`, `%LOCALAPPDATA%\1C\1cv81\` и т.д.

### 6.3 Источники данных

- `%APPDATA%\1C\1CEStart\ibases.v8i` — список баз, поле `LocalCache` → UUID папки
- `%APPDATA%\1C\1CEStart\1CEStart.cfg` — общие списки баз (CommonInfoBases)
- `{uuid}\1CV8Clnt.cfg` — путь/сервер базы для каждой UUID-папки

### 6.4 Структура дерева — вид по пользователю (по умолчанию)

```
Кэш                                        [40.2 GB]
├── ivanov                                 [12.4 GB]
│   ├── Бухгалтерия_Прод                   [ 8.1 GB]
│   ├── ЗУП_Тест                           [ 3.2 GB]
│   └── [мёртвые папки]                    [ 1.1 GB]
├── petrov                                 [ 8.0 GB]
│   ├── Бухгалтерия_Прод                   [ 6.0 GB]
│   └── ЗУП_Тест                           [ 2.0 GB]
└── sidorov                  [исключён]    [ 0.0 GB]
```

### 6.5 Структура дерева — вид по базе (Tab)

```
Кэш                                        [40.2 GB]
├── Бухгалтерия_Прод                       [14.1 GB]
│   ├── ivanov                             [ 8.1 GB]
│   └── petrov                             [ 6.0 GB]
├── ЗУП_Тест                               [ 5.2 GB]
│   ├── ivanov                             [ 3.2 GB]
│   └── petrov                             [ 2.0 GB]
└── [мёртвые папки]                        [ 1.1 GB]
    └── ivanov                             [ 1.1 GB]
```

### 6.6 Порядок выполнения операции удаления

1. Выделение узлов (пробел)
2. `D` — Dry Run: список файлов и папок, суммарный объём (Magenta)
3. Подтверждение (уровень зависит от объёма операции)
4. Повторная проверка запущенных процессов 1С
5. Перемещение в Backup (если включён) → удаление
6. Отчёт: удалено папок, освобождено места, пропущено файлов

## 7. Безопасность операций

### 7.1 Защищённые файлы (хардкод, нельзя удалить)

| Маска | Описание |
|---|---|
| `*.lic` | Лицензионные файлы 1С |
| `*.pfl` | Профиль пользователя 1С |
| `*.usr` | Настройки пользователя |
| `1CV8Clnt.flt` | Фильтры клиента |
| `*.1CD` | Файл файловой базы данных |
| `*.dbf` | Файлы таблиц базы данных |

Защита реализована в `SafeDelete.cs` и не может быть отключена через UI.

### 7.2 Поиск зависших блокировок

Отдельное действие: поиск и удаление файлов `*.1CD.LCK` — зависших блокировок файловых баз.

Дополнительно: для файловых баз в контекстном меню доступен запуск `chkfl.exe /F` (Phase 2).

### 7.3 Dry Run

- Доступен перед любой деструктивной операцией
- Показывает список, суммарный объём, защищённые файлы которые будут пропущены
- Цвет Magenta, пометка `[ПРЕДПРОСМОТР — ничего не удаляется]`

### 7.4 Backup

- **Выключен по умолчанию** — включается осознанно через настройки
- Путь задаётся в реестре: `HKCU:\Software\Clinkon1C\Settings\BackupPath`
- По умолчанию при включении: `C:\Temp\Clinkon1C\Backup\{ГГГГ-ММ-ДД_ЧЧ-ММ-СС}\`
- Предупреждение если на диске мало места перед включением
- Политика автоудаления: старше N дней ИЛИ при остатке места на диске < 10%

### 7.5 Обработка занятых и недоступных файлов

- При ошибке доступа — определение блокирующего процесса через **Windows Restart Manager API** (`RestartManagerHelper.cs`)
- Вывод в UI: `[Заблокирован: splwow64.exe (PID: 4512)]` с предложением завершить процесс
- Если процесс не определён — retry: 3 попытки с интервалом 2 сек
- При битых ACL — попытка `takeown` + `icacls` (только с правами администратора)
- Неудалённые файлы — в лог `WARN` и сводка после операции

### 7.6 Длинные пути

Все файловые операции через `System.IO` с нативной поддержкой путей > 260 символов (.NET 8 out of the box).

### 7.7 Символические ссылки и Junction

При обнаружении symlink или junction — удаляется только ссылка, содержимое не трогается.

## 8. Лог операций утилиты

- Путь: `C:\ProgramData\Clinkon1C\logs\clinkon_ГГГГ-ММ-ДД.log`
- Уровни: `INFO`, `WARN`, `ERROR`
- Что логируется: пользователь запустивший утилиту, выбранные операции, удалённые объекты, размеры, ошибки ACL, блокирующие процессы, пропущенные файлы
- Ротация: хранить не более 90 дней
- Запись критических событий в **Windows EventLog** (для SIEM-систем)

---

## 9. Исключения

Хранятся в реестре: `HKCU:\Software\Clinkon1C\Exclusions`

- Исключить конкретного пользователя (весь его кэш, логи, шаблоны)
- Исключить конкретную базу (по имени или UUID)
- В дереве исключённые узлы: цвет DarkCyan, пометка `[исключён]`
- Управление исключениями — отдельный пункт меню утилиты

## 10. Модуль Логи (Phase 2)

### 10.1 Структура дерева

```
Логи
├── Платформа                              (клиентские логи по пользователям)
│   ├── ivanov                             [ 0.5 GB]
│   └── petrov                             [ 0.7 GB]  ← [!] порог 1 GB
├── Сервер 1С                              (только если мы на сервере)
│   ├── Агент сервера                      [ 1.5 GB]
│   └── Temp службы                        [ 0.3 GB]
└── Журналы регистрации
    ├── Файловые базы
    │   └── Бухгалтерия_Файл
    │       ├── 2025-03                    [120 MB]
    │       ├── 2025-04                    [340 MB]
    │       └── 2025-05                    [ 89 MB]  ← текущий
    └── Серверные базы
        ├── Бухгалтерия_Прод              (доступна — мы на сервере)
        └── ЗУП_Прод                      [недоступно: server2]
```

### 10.2 Пороги подсветки

| Тип | Порог по умолчанию | Причина |
|---|---|---|
| Кэш баз | 5 GB | Норма для активного пользователя |
| Логи платформы | 1 GB | Признак забытого Технологического Журнала |
| Журналы регистрации | 5 GB | Зависит от активности базы |

Пороги настраиваются в реестре.

### 10.3 Особенности .lgd (SQLite)

Файлы `.lgd` — SQLite-базы журналов регистрации. Удаление по дате внутри файла не поддерживается. Варианты действия:
- Удалить файл за период целиком
- Пропустить с отображением размера

### 10.4 Определение расположения серверных журналов

- Реестр: `HKLM:\SOFTWARE\1C\1Cv8\`
- Конфиг агента: `1cv8wsrv.lst`
- Если база на другом сервере — узел серым: `[недоступно: имя_сервера]`

### 11.2 Вид по конфигурации (Tab)

Показывает у кого какая версия — для быстрой очистки старых версий у всех пользователей сразу.

### 11.3 Объединение шаблонов

- Целевая папка выбирается из динамического списка или вводится вручную
- Дедупликация по **SHA1-хешу** (не по имени файла)
- При совпадении имени с разным хешем — запрос у пользователя
- После переноса — опциональная символическая ссылка из исходных папок на целевую

## 12. Phase 3 — Процессы и RAC/RAS

### 12.1 Клиентские процессы

| Процесс | Тип |
|---|---|
| `1cv8.exe` | Толстый / тонкий клиент |
| `1cv8c.exe` | Конфигуратор |

Отображение: PID, пользователь, тип клиента, база (из CommandLine процесса).

Действия: завершить выбранный / все у пользователя / все (мягко или жёстко).

### 12.2 Серверные компоненты

Управление через утилиту `rac` (если найдена на сервере):
- Просмотр сессий кластера
- Завершение конкретной зависшей сессии (graceful через rac, не kill процесса)
- Просмотр блокировок данных

Управление службами (стоп / старт / рестарт) — крайняя мера, с подтверждением `ОСТАНОВИТЬ`.

## 18. Поддержка Windows 7 — два билда

### 18.1 Стратегия

Выпускаются два отдельных исполняемых файла из одной кодовой базы:

| Файл | Платформа | Требования | Аудитория |
|---|---|---|---|
| `Clinkon1C-x64.exe` | .NET 8, self-contained | Windows 10 / Server 2012 R2+ | RDP-серверы, современные ПК |
| `Clinkon1C-legacy.exe` | .NET Framework 4.8 | Windows 7 SP1+, Framework 4.8 | Старые рабочие места |

Оба файла публикуются в одном GitHub Release. Пользователь выбирает нужный.

### 18.2 Проверка .NET Framework при запуске legacy-версии

При запуске `Clinkon1C-legacy.exe` первым делом проверяется наличие .NET Framework 4.8 через реестр:

```
HKLM:\SOFTWARE\Microsoft\NET Framework Setup\NDP\v4\Full
Значение Release >= 528040  →  Framework 4.8 установлен
```

### 18.3 Поведение если Framework не найден

```
╔══════════════════════════════════════════════════════════════╗
║  Требуется .NET Framework 4.8                                ║
║                                                              ║
║  Для работы утилиты необходим Microsoft .NET Framework 4.8.  ║
║  Он не обнаружен на этом компьютере.                         ║
║                                                              ║
║  [1] Открыть страницу загрузки на сайте Microsoft            ║
║  [2] Выход                                                   ║
╚══════════════════════════════════════════════════════════════╝
```

При выборе `[1]` — открывается браузер по умолчанию с прямой ссылкой на загрузку Framework 4.8 с сайта Microsoft.

Автоматическая установка Framework не производится — пользователь принимает решение самостоятельно.

### 18.4 Ограничения legacy-версии

- Требует установленного .NET Framework 4.8 на машине
- Framework 4.8 устанавливается один раз администратором
- Минимальная ОС: Windows 7 SP1 с обновлениями
